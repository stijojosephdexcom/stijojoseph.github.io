Interrupts & ISR

Interrupts let hardware events steal the CPU instantly — milliseconds matter.

01. What it is

An interrupt is an asynchronous signal that forces the CPU to pause whatever it's doing, save its state, and jump to an Interrupt Service Routine (ISR). When the ISR returns, execution continues exactly where it left off.

Interrupts are how an MCU reacts to the real world: a button press, a UART byte arriving, a DMA transfer finishing, a timer overflow.

02. How it works

On ARM Cortex-M the Nested Vectored Interrupt Controller (NVIC) arbitrates up to 240 interrupt sources with configurable priority. When a peripheral asserts its interrupt line:

  1. The NVIC compares the new IRQ's priority against the currently executing one.
  2. If it's higher, the core tail-chains — pushes R0–R3, R12, LR, PC, xPSR onto the stack in 12 cycles.
  3. It loads the ISR address from the vector table at the start of Flash.
  4. The ISR executes; on return the stack is unwound and normal flow resumes.

Best practice: keep ISRs short. Set a volatile flag and let the main loop or an RTOS task do the heavy lifting (deferred processing).

04. Real-world examples

  • Button debouncing — EXTI fires on edge, ISR sets flag, main loop confirms after 20 ms.
  • UART RX — interrupt per byte, store in ring buffer, signal a parser task.
  • Motor encoder — quadrature interrupts on both A and B lines update a position counter.
  • Touch screen — controller asserts INT line when finger detected; ISR reads coordinates over I²C.

05. Code snippet

C — STM32 EXTI handler 
volatile uint8_t g_buttonFlag = 0;

/* Configure PA0 as EXTI rising-edge interrupt */
void EXTI0_IRQHandler(void) {
    if (__HAL_GPIO_EXTI_GET_IT(GPIO_PIN_0)) {
        g_buttonFlag = 1;          /* defer work */
        __HAL_GPIO_EXTI_CLEAR_IT(GPIO_PIN_0);
    }
}

void main_loop(void) {
    while (1) {
        if (g_buttonFlag) {
            g_buttonFlag = 0;
            handleButtonPress();
        }
        otherWork();
    }
}

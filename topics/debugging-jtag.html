Debugging & JTAG / SWD

JTAG and SWD give you a microscope into the running CPU — set breakpoints, inspect registers, single-step instructions.

01. What it is

JTAG (IEEE 1149.1) was originally a manufacturing test interface (boundary scan) but became the universal back-door for debugging embedded CPUs. ARM later defined a slimmer 2-pin variant called SWD (Serial Wire Debug) with the same capability and far fewer pins.

Through the debug port you can halt the CPU at any time, read and write any memory or register, set hardware breakpoints, and single-step instructions.

02. How it works

On Cortex-M the debug logic lives in three blocks:

  • DAP (Debug Access Port) — turns SWD/JTAG transactions into AHB bus transactions.
  • FPB (Flash Patch & Breakpoint) — up to 6 hardware breakpoints in Flash.
  • DWT (Data Watchpoint & Trace) — watchpoints, cycle counting, instruction trace.

An IDE's debugger talks to a probe (ST-Link, J-Link, CMSIS-DAP) which in turn drives JTAG/SWD lines and reads/writes the debug registers. Step → set the single-step bit, run, halt, refresh GUI.

04. Real-world examples

  • ST-Link / J-Link probes — the USB debug pods every embedded engineer owns.
  • OpenOCD + GDB — the open-source toolchain used by Eclipse, VSCode and CLion.
  • Segger SystemView — uses SWO trace pin to visualise RTOS task switches in real time.
  • Boundary scan — chip manufacturers use JTAG to test PCB connectivity at production.
